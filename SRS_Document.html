Software Requirements Specification (SRS) v1.0

1. Introduction

1.1 Purpose

This Software Requirements Specification (SRS) document describes the functional and non-functional requirements for the SGP Platform, a comprehensive e-commerce and interior design management system. It serves as the authoritative reference for developers, stakeholders, and quality assurance teams throughout the development lifecycle.

1.2 Scope

The SGP Platform is a full-stack Next.js web application designed to facilitate:

  • Product catalog management and e-commerce transactions
  • Premium interior design booking and consultation services
  • AI-powered room matching and design recommendations
  • User authentication and profile management
  • Shopping cart and order management
  • Administrative dashboard for system management
  • Message and communication system
  • Gallery and inspiration content management

2. Overall Description

2.1 Product Perspective

SGP Platform is an independent, modern web application that operates as a Software-as-a-Service (SaaS) solution. It integrates with multiple third-party services including:

  • MongoDB: Primary data persistence layer
  • Cloudinary: Image hosting and optimization
  • Nodemailer: Email communication service
  • Stripe/PayPal: Payment processing (future implementation)
  • WhatsApp Business API: Business communications
  • Sentry: Error tracking and monitoring
2.2 User Classes and Characteristics

Designer Interior design specialists providing consultation View appointments, provide design consultations, manage portfolio (future)
Guest Unauthenticated users browsing the platform View products, explore inspirations, contact support

  • Authentication: NextAuth.js for session management with JWT-based tokens
  • Security: HTTPS only, CORS protection, rate limiting, CSP headers
  • Performance: Lazy loading, image optimization, code splitting
  • Scalability: Stateless design for horizontal scaling
  • Compliance: GDPR data protection, payment PCI-DSS compliance (future)

  • Users have modern web browsers with JavaScript enabled
  • Internet connectivity is available for all transactions
  • Third-party services (MongoDB, Cloudinary) remain available
  • Email service is operational for notifications
  • Payment gateway integration will be implemented before production

5. Functional Requirements

5.1 User Authentication Requirements
FR-1.1: System shall provide user registration with email validation
FR-1.2: System shall enforce strong password requirements (min 8 characters, mixed case, numbers, symbols)
FR-1.3: System shall hash passwords using bcryptjs before storage
FR-1.4: System shall provide login with email and password
FR-1.5: System shall maintain user sessions with automatic timeout after 30 minutes of inactivity
FR-1.6: System shall provide password reset functionality via email
FR-1.7: System shall enforce role-based access control (Customer, Admin, Designer)
5.2 Product Management Requirements
FR-2.1: System shall allow admins to create products with name, description, price, and images
FR-2.2: System shall support product categorization and filtering
FR-2.3: System shall display product details including specifications and dimensions
FR-2.4: System shall maintain product inventory (future: real-time availability)
FR-2.5: System shall provide product search functionality
5.3 Shopping Cart Requirements
FR-3.1: System shall allow users to add/remove products from cart
FR-3.2: System shall persist cart data across sessions
FR-3.3: System shall calculate subtotal, taxes, and total dynamically
FR-3.4: System shall support discount codes and coupon application
FR-3.5: System shall provide cart abandonment recovery via email
5.4 Order Management Requirements
FR-4.1: System shall create orders from checkout process
FR-4.2: System shall track order status (Pending, Confirmed, Shipped, Delivered)
FR-4.3: System shall send order confirmation email to customer
FR-4.4: System shall generate and email invoices
FR-4.5: System shall maintain order history accessible to users
5.5 Booking Management Requirements
FR-5.1: System shall allow customers to book premium design consultations
FR-5.2: System shall prevent double-booking of appointments
FR-5.3: System shall send appointment reminders via WhatsApp
FR-5.4: System shall allow admins to approve/reject bookings
FR-5.5: System shall track booking status and history
5.6 Wishlist Requirements
FR-6.1: System shall allow users to add/remove items from wishlist
FR-6.2: System shall persist wishlist data per user account
FR-6.3: System shall allow sharing of wishlists with other users
5.7 Admin Dashboard Requirements
FR-7.1: System shall provide admin dashboard with key metrics (orders, revenue, users)
FR-7.2: System shall display sales analytics with charts and graphs
FR-7.3: System shall allow data export to Excel/PDF formats
FR-7.4: System shall provide user management interface

6. Non-Functional Requirements

6.1 Performance Requirements
  • Page Load Time: Maximum 3 seconds on 3G network (FCP)
  • API Response Time: 95th percentile under 200ms for read operations, 500ms for write operations
  • Database Query Performance: All queries must complete within 100ms on indexed columns
  • Concurrent Users: Support minimum 1000 concurrent users without degradation
  • Throughput: Minimum 100 requests/second per instance
  • Image Optimization: All images served at appropriate sizes and formats (WebP when supported)
6.2 Security Requirements
  • Encryption: All data in transit encrypted with TLS 1.3
  • Authentication: Multi-layer authentication with JWT and session tokens
  • Authorization: Strict role-based access control at all endpoints
  • SQL Injection Prevention: Parameterized queries and ORM usage
  • XSS Prevention: Content Security Policy headers and input sanitization
  • CSRF Protection: CSRF tokens for state-changing operations
  • Rate Limiting: API rate limiting to prevent abuse
  • Data Privacy: GDPR compliance with user data deletion and export capabilities
  • Password Security: Bcrypt hashing with salt rounds ≥ 10
  • Secure Headers: X-Content-Type-Options, X-Frame-Options, X-XSS-Protection
6.3 Availability & Reliability
  • Uptime SLA: 99.5% availability (target 4.35 hours/month max downtime)
  • Recovery Time Objective (RTO): 15 minutes maximum
  • Recovery Point Objective (RPO): 1 hour maximum data loss acceptable
  • Backup Strategy: Daily incremental backups with weekly full backups
  • Failover: Automatic failover to backup infrastructure within 5 minutes
  • Monitoring: Real-time monitoring with alerting for critical issues
6.4 Scalability Requirements
  • Horizontal Scaling: Stateless architecture supporting horizontal scaling
  • Database Scaling: MongoDB replica sets for read scaling
  • CDN Integration: Static assets served from CDN for global performance
  • Caching Strategy: Multi-layer caching (browser, CDN, application, database)
  • Load Balancing: Automatic load distribution across servers
6.5 Usability Requirements
  • Interface Clarity: Intuitive navigation with clear visual hierarchy
  • Accessibility: WCAG 2.1 AA compliance for users with disabilities
  • Mobile Responsiveness: Full functionality on devices from 320px width
  • Error Handling: Clear error messages with actionable solutions
  • Help & Documentation: In-app help, FAQs, and user guides
  • Dark Mode: Optional dark theme for comfortable viewing
6.6 Maintainability Requirements
  • Code Documentation: JSDoc comments for all functions and complex logic
  • Testing Coverage: Minimum 80% code coverage with unit and integration tests
  • Version Control: Git with semantic versioning for releases
  • CI/CD Pipeline: Automated build, test, and deployment
  • Logging: Comprehensive logging with multiple levels (debug, info, warn, error)
  • Monitoring: Application performance monitoring and error tracking
6.7 Compliance Requirements
  • GDPR compliance for user data handling
  • PCI-DSS compliance for payment data (future implementation)
  • WCAG 2.1 for accessibility
  • Terms of Service and Privacy Policy compliance

7.4 Design Patterns
  • MVC Pattern: Separation of concerns with Models (MongoDB schemas), Views (React components), Controllers (API routes)
  • Context API Pattern: Global state management for auth, cart, and dreams contexts
  • Middleware Pattern: Authentication, CORS, rate limiting, request logging middleware
  • Factory Pattern: Dynamic component creation based on data types
  • Observer Pattern: Event-driven architecture for real-time updates
  • Repository Pattern: Data access abstraction through Mongoose models

Precondition: Customer is on product page
Main Flow:
  1. System displays product catalog
  2. Customer filters by category, price range
  3. Customer sorts by popularity, price
  4. Customer views product details

Actor: Customer
Precondition: Customer has items in cart, authenticated

  1. Customer proceeds to checkout
  2. System displays order summary
  3. Customer enters/confirms shipping address
  4. Customer selects payment method
  5. System processes payment (future)

  3. Customer selects date and time
  4. System checks availability
  5. Customer provides service details (room type, budget)
  6. System creates booking request
  7. Admin reviews and approves booking
  8. System sends confirmation to customer
